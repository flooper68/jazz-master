# How the product works

Jazz Master is a web app that helps guitarists get better at jazz — a structured practice companion, not a chord-chart reference. The core promise (VIS-001) is **zero-tension practice**: open the app, pick up the guitar, and play. The app has already decided today's session from your level, goals, and history, and guides you through it exercise by exercise. It targets intermediate guitarists learning the jazz language — beginners in jazz, not on the instrument.

## The practice flow

The intended end-to-end session, once the guided-practice slice lands (strategy/goals.md "Now" item 3):

1. **Dashboard** shows today's plan, generated by the adaptive planner from the signed-in user's server-owned profile plus Postgres session history (EPIC-011, EPIC-012, EPIC-013).
2. **Guided lesson** runs the plan exercise by exercise — drills like "play this ii–V–I in Bb, now in Eb, now with drop 2 voicings" — rendered guitar-first: fretboards, grip diagrams, and — for scale/arpeggio exercises since 2026-07-07 (TASK-038) — staff notation with tablature via VexFlow 5 behind a project-owned `<Notation>` component, opted in per exercise through a `'notation'` display hint (EPIC-008, EPIC-009; ADR-010). Since TASK-048, that score defaults larger, reserves its layout space, lets the player choose staff, TAB, or both with a persisted preference, and can expand into an in-run focus mode with grading controls. Since TASK-047, each exercise panel can also play the exercise notes with sampled guitar timbre, loop them with click/count-in, and remember a slower practice tempo per exercise. TASK-069 now syncs notation mode, scoring tolerance, and per-exercise tempo through the signed-in user's Postgres data.
3. **Grading**: self-graded first; recording with in-browser pitch/timing scoring — VIS-001's riskiest bet — is researched **staged-go** (RES-014): monophonic offline-after-take analysis for v1, chord scoring staged later. The recommended real-guitar spike was skipped by accepted owner risk (NOTE-010), so EPIC-010 proceeds from RES-014 defaults and synthesized-test evidence. The runner now has the capture UI (mic permission, input meter, count-in, in-session replay) and, since TASK-043, decodes a stopped take locally, shows a 0-100 score with pitch/timing/completeness and per-note verdicts, and persists score metadata with the graded exercise while leaving raw audio transient. NOTE-012 moves human-only browser/device mic coverage to QA/product review.
4. **History**: the session lands in practice history; the dashboard shows streaks, scores, keys mastered — momentum (EPIC-012).

Content exists to feed drills and lessons, not as browsable reference — "the app decides, you play" is the differentiator.

## Module pillars

Each pillar maps to an epic (VIS-001): foundation (EPIC-001), chord voicings (002), progression trainer (003), practice tools/metronome (004), repertoire (005), ear training (006), curriculum & lessons (008), notation & tabs (009), recording & scoring (010), adaptive planner (011), dashboard & history (012), plus the multi-app platform track (013). Ways-of-working (007) is process, not product. Non-goals for now: social features, native mobile, notation *editing*, bundled copyrighted material, and account features beyond the Clerk sign-in needed for user-owned practice data.

## How it works under the hood

The app is in a persistence migration. Signed-out visitors now start from a
polished Astro-owned public surface: `/` explains the jazz-practice loop with
practice-board-style visuals and CTAs, while `/sign-in` and `/sign-up` reuse the
same public shell around Clerk's prebuilt widgets. Clerk protects `/app/*`, and
signed-out app entry lands on Jazz Master's own `/sign-in` page, with `/sign-up`
as the local account-creation counterpart. Exact and rest-parameter Astro routes render Clerk's
prebuilt auth UI, so password recovery, MFA/2FA, and required session tasks
remain Clerk configured rather than custom app code. Profile/onboarding data,
sessions, grades, and score metadata now use protected tRPC plus normalized
Postgres tables keyed by Clerk user ID. Today's plan is no longer a stored
browser snapshot: protected `planner.today` recomputes it from the server-owned
profile, server-owned session history, curriculum, and the browser-local plan
date. TASK-069 moved notation mode, scoring tolerance, and per-exercise tempo
over the same authenticated tRPC/Postgres boundary; authored tempo, notation
`both`, and scoring `standard` remain the defaults when no row exists.
`/app/profile` now explains account sync and no longer offers local JSON backup
actions. TASK-070 removed the obsolete JSON serializer, parser, and public API;
TASK-071 then removed the generic browser-persistence wrapper and compatibility
exports. Client-safe contracts now live in `src/appData/`, while durable data
stays behind tRPC/Postgres. Legacy local payloads are ignored. ADR-012 is the
implemented architecture: Clerk owns identity,
browser code talks to typed tRPC procedures,
server code owns Drizzle/Postgres access, and Postgres becomes the source of
truth for long-run app data. Existing local browser data is intentionally
discarded rather than migrated, and product source has no browser persistence
layer.

Domain logic — notes, intervals, chord spelling, scales, arpeggios, fretboard
positions, with enharmonics correct by construction — lives in the pure
`@jazz-master/theory` package; curriculum content (exercises and lessons in
`apps/web/src/content/`) references theory constructs rather than hard-coding
notes, so a lesson is "the Bb major scale in position II", not a note list;
pages compose reusable components (Fretboard, ChordDiagram) which render what
theory computes. The shell is now hybrid (EPIC-013 / ADR-006, first slice
shipped as TASK-021): Astro serves the signed-out landing page at `/` and hosts
the whole React practice app as a client-only SPA island under `/app/*` — so
every product route above now carries the `/app` prefix (`/app/practice`,
`/app/history`, …). The build targets Cloudflare Workers; Clerk middleware
protects `/app/*`, app-hosted `/sign-in` and `/sign-up` pages own auth entry,
and the app shell exposes Clerk's account control. A typed tRPC API surface now
exists (TASK-023/TASK-066): `/trpc/health`, protected
procedure support via `auth.clerkUserId`, `users.ensure` for the DB-backed
app-user anchor, `profile.get`/`profile.save`, `sessions.list`/`sessions.upsert`,
and `planner.today` for server-computed daily planning, plus protected
preference reads/writes.
Since TASK-055/TASK-069, server-side Postgres work has Drizzle migration
infrastructure: local development uses the repo-owned Compose database,
generated migrations live in the dedicated Railway migration app, deployment
migrations run from that `apps/migration` service with Railway-owned
`DATABASE_URL`, a minimal `users` table keyed directly by Clerk user ID scopes
app data, normalized profile tables store onboarding/profile data, and
normalized session/score tables store practice history and per-note score
details, `user_preferences` stores the two scalar settings, and
`play_along_tempos` stores one clamped override per user/exercise. The earlier smoke-only
`mockPractice` scaffolding has been removed; `dbSmoke` remains as a public
observability probe for deployed database connectivity.
Cloudflare Workers Builds does not receive database credentials, and the Worker
request path does not run migrations. Deployment is CI-only per ADR-009 —
Cloudflare Workers Builds auto-deploys the dev environment at
https://jazz-master.premysl-ciompa.workers.dev on every push to `main` (live
since 2026-07-07, TASK-024), no deploy credential exists anywhere agents can
reach, and there is no production environment — deliberately: the dev URL is the
product's home, and the parked production task was abandoned outright (TASK-036,
owner decision 2026-07-07, NOTE-008).

## Built today vs pending (as of 2026-07-10)

Done: app shell with routing and sidebar, the theory core, fretboard and chord-diagram components, and the original typed browser-persistence layer — EPIC-001 (foundation) is complete, and ADR-012/TASK-071 have since retired that layer in favor of Clerk/Postgres. EPIC-008 (curriculum & lessons) is complete: the exercise/lesson content model (TASK-011), the first lesson pack (TASK-012 — 10 scales/arpeggios lessons across 3 levels), and the guided practice runner (TASK-013) — from the Practice page a user starts a lesson, is guided exercise by exercise (fretboard positions, countdown, got-it/shaky/missed self-grades), and the session — completed or abandoned — persists as a server-owned `PracticeSession` record through protected tRPC. EPIC-011 (adaptive planner) is complete and migrated: first-run onboarding or skip writes a Clerk/Postgres-backed `PracticeProfile`, `/profile` edits it through tRPC, and `/practice` plus the dashboard read a server-computed Today's plan generated from profile, lesson metadata, session history, and date. EPIC-012 (dashboard & history) is complete: the practice-history page (TASK-018) makes every persisted session findable at `/history` — grouped by day newest-first, filterable by area and time range, with grade tallies, incomplete markers, session-level machine scores, and drill-down to per-exercise self-grades/scores — and the dashboard v1 (TASK-019) replaces the `/` stub with the product's front door: today's plan with reasons and a one-click Start into the runner, current streak, minutes-this-week against the profile budget, and per-area status with needs-attention callouts (trend-over-time visuals wait for broader score history). EPIC-009 (notation & tabs) is complete as of 2026-07-07: every lesson-pack scale/arpeggio exercise renders an aligned staff + TAB pair below the fretboard in the runner (TASK-037 component, TASK-038 integration), VexFlow lazy-loads only when a notation exercise appears, and the bundle trim (TASK-039) cut that lazy chunk from 692 to 389 KB gzip via the single-music-font `vexflow/bravura` entry. TASK-048 added the post-epic readability layer: larger default score scaling, reserved score space, keyboard-scrollable overflow, staff/TAB/both controls, a persisted notation display preference, focus mode, and a smoke assertion that real SVG score content rendered. EPIC-014 (play-along) is complete: the runner lazy-loads sampled guitar playback only when Play is pressed, loops the exercise with click/count-in, lets the player slow the tempo up to the authored target BPM, and remembers that tempo per exercise. EPIC-010's monophonic v1 loop is implemented through TASK-043: the runner can request mic access, show a level meter, count in, record, replay, locally analyze the stopped take, show per-note score feedback, persist score metadata, and discard raw audio. NOTE-012 moved human-only desktop/mobile browser mic coverage to QA/product review rather than task completion gates; real-guitar plausibility remains something QA/dogfooding must watch. Everything user-facing beyond that — moving drills and chord close-enough scoring — remains pending: the other practice modules are stub pages. The guided-practice vertical slice (onboard → plan → practice → history → dashboard) is finished, and the notation/audio-scoring de-risking research (strategy/goals.md goal 4) is done. The EPIC-013 platform migration is implemented through TASK-071: the app runs on Cloudflare Workers, with local Postgres, Drizzle migrations, the Railway migration service, Clerk auth, the Clerk-keyed user anchor, server-owned profile/sessions/planning/preferences, and no product browser-persistence layer. ADR-012 is implemented; only TASK-072's final regression pass remains.

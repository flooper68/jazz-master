The project will likely need Storybook plus dedicated UI skills/process guidance
for implementing and testing UI design work. This would give agents a concrete
place to build, inspect, and regression-test reusable UI states rather than
validating every design change only inside full app flows.

Why it might matter: Jazz Master is accumulating reusable practice UI and
multiple product areas. Without a component-level design workflow, visual
quality, edge states, responsiveness, and accessibility can depend too heavily
on ad hoc browser checks during feature work.

## Product framing

Current condition: UI implementation and verification happen through the app,
project processes, and task-specific tests, with no dedicated Storybook-style
component workbench.

Desired condition: UI design work has a clear process, reusable agent guidance,
component scenarios, and repeatable checks for visual states before integration.

Affected user/workflow: agents implementing product UI; indirectly, guitarists
using practice workflows that depend on polished, reliable interface states.

Evidence: owner identified Storybook plus UI skills/process as a desired
long-term project capability.

Validation need: research/spike
